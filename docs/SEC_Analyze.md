对 `MagpieAuth` 项目的完整安全风险评估已完成。以下是基于代码逻辑和架构的安全性分析报告，不涉及对代码的任何修改：

### 🔴 1. 高危风险 (High / Critical Risks)

**1.1 内部主密钥 (IMK) 存在本地明文存储漏洞**

* **代码位置:** `src-tauri/src/crypto.rs` (`store_imk` / `retrieve_imk`)
* **问题分析:** 系统用于加密保险库中所有密码和 TOTP 密钥的 256 位内部主密钥 (IMK)，直接以明文的形式被写入到用户数据目录下的 `.imk` 文件中（如 Windows 下的 `%APPDATA%\MagpieAuth\.imk`）。
* **安全风险:** 任何拥有该设备文件读取权限的恶意软件或本地攻击者，都可以轻易读取该密钥文件。结合同样存储在本地的 `vault.db`，攻击者能够瞬间解密受害者的所有密码和 TOTP 令牌，导致底层的高级加密 (AES-256-GCM) 形同虚设。
* **修复建议:** 必须废弃明文文件存储的方式，改用操作系统原生的安全凭据管理器（如 Windows Credential Manager、macOS Keychain、Linux Secret Service），可以通过 Rust 的 `keyring` 库来实现安全存储。

**1.2 非 Windows 平台的系统认证保护被硬编码绕过**

* **代码位置:** `src-tauri/src/auth.rs` (`verify_user`)
* **问题分析:** 代码使用 `#[cfg(not(windows))]` 宏，在非 Windows 平台（如 macOS 或 Linux）上，将系统认证接口 `verify_user()` 的实现直接硬编码为返回 `Ok(true)`。
* **安全风险:** 这是一个典型的开发者“测试桩代码 (Stub)”残留。如果应用在 Mac 或 Linux 上构建运行，任何本该被系统级生物识别（指纹/面容）或系统密码拦截的敏感操作（如查看密码、导出数据）都会被静默放行，没有任何验证环节。
* **修复建议:** 引入跨平台的本地认证依赖，例如在 macOS 接入 `LocalAuthentication` 框架实现 Touch ID/密码认证，或在检测不到系统认证能力的设备上强制要求用户设置高强度主密码 (Master Password)。

**1.3 缺失内容安全策略 (CSP)**

* **代码位置:** `src-tauri/tauri.conf.json`
* **问题分析:** Tauri 配置文件中显式配置了 `"csp": null`，即完全禁用了 Web 视图的内容安全策略。
* **安全风险:** 现代前端极其容易受到跨站脚本攻击 (XSS)。一旦应用中的某些字段（如条目标题、账号名称）没有被正确转义并被渲染，攻击者可以执行恶意 JavaScript。在没有 CSP 保护的情况下，恶意脚本可以轻易调用 Tauri API 窃取所有保险库数据或与远程服务器进行隐蔽通信。
* **修复建议:** 配置严格的 CSP（例如限制 `default-src 'self'`，禁止 `unsafe-inline` 和 `unsafe-eval`，阻断外部网络请求），遵循 Tauri 官方的安全最佳实践。

### 🟠 2. 中危风险 (Medium Risks)

**2.1 AES-GCM 加密缺失附加身份验证数据 (AAD)**

* **代码位置:** `src-tauri/src/crypto.rs` (`encrypt_field` / `decrypt_field`)
* **问题分析:** 尽管代码正确选用了优秀的 AES-256-GCM 认证加密算法，但在加密和解密时，未将数据所属的上下文（如条目的 `id` 或 `title`）作为附加认证数据 (AAD) 绑定进去。
* **安全风险:** 存在**密文替换攻击 (Ciphertext Swapping)** 风险。攻击者可以直接操作 `vault.db` 数据库，将条目 A（如一个无足轻重的测试账号）的加密密码替换为条目 B（如网银账号）的加密密码。由于没有 AAD 校验，解密依然会成功。
* **修复建议:** 在调用 GCM 的加密/解密方法时，将该条目对应的唯一 `id` 作为 AAD 混入，确保加密内容只能被对应的数据库行正确解密。

### 🟡 3. 低危风险与架构缺陷 (Low Risks & Architecture Issues)

**3.1 敏感信息在内存中明文驻留且缺乏清理机制**

* **问题分析:** 在 `crypto.rs` 中，后端虽然对生成 IMK 时使用了 `Zeroizing` 以防内存泄露，但解密密码发送给前端的过程中（如 `String::from_utf8_lossy`）产生了许多未归零的 String 副本。同时，明文到达前端后长期存留在 React 的状态 (JavaScript 堆内存) 中。
* **安全风险:** 恶意软件通过对应用程序进行内存转储 (Memory Dump) 依然可以提取出明文密码。这受限于 Tauri / Electron 等 Webview 架构，很难完全根除。
* **修复建议:** 尽可能缩短明文在前端显示的生命周期；例如使用只读组件，用户复制后或者离开视线后立即覆盖、卸载相关组件以协助垃圾回收。

**3.2 导出备份功能依赖的弱密码风险**

* **代码位置:** `src-tauri/src/commands.rs` (`export_vault`)
* **问题分析:** 导出时直接将用户输入作为密码，结合 Argon2id 派生密钥并加密导出文件。
* **安全风险:** 如果用户为了图方便输入了很弱的导出密码（比如 `123456`），那么导出的备份文件极易遭到离线字典攻击。
* **修复建议:** 在前端强制校验导出密码的复杂度（例如必须包含大小写字母、数字及特殊字符，长度至少 8 位以上），以保证备份文件的安全性。

---

**最终结论：** 当前项目具有基本的密码学框架（Argon2id 加盐哈希、AES-GCM 加密）， **但是在关键的信任根管理上存在逻辑致命伤（即密钥文件明文裸存）** 。在将其用于实际的个人数据存储之前，首要任务是必须将 IMK (内部主密钥) 的存取交由操作系统的系统级安全凭据库接管，并且补齐跨平台系统认证与 CSP 安全策略。
